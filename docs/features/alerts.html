Alert Management

Alerts are the core of Pulsar's incident management system. They represent events that require attention from your team, such as system failures, performance issues, or security threats.

Alert Lifecycle

Alerts in Pulsar follow a defined lifecycle:

Open --> Acknowledged --> Closed
  |          |
  |          v
  +-----> Snoozed ----+
                      |
                      v
                   Reopened
Status Description
Open New alert that needs attention
Acknowledged Someone is working on it
Snoozed Temporarily silenced until a specified time
Closed Issue resolved or no longer relevant

Priority Levels

Alerts have priority levels that determine their urgency:

  • Critical (P1) - Immediate action required, major service impact
  • High (P2) - Urgent, significant impact
  • Medium (P3) - Important but not urgent
  • Low (P4) - Informational, can be addressed later

Creating Alerts

Via Web UI

  1. Navigate to Alerts in the sidebar
  2. Click Create Alert
  3. Fill in the title, description, priority, and optional tags
  4. Assign to a team or user (optional)
  5. Click Create

Via API

curl -X POST http://localhost:8081/api/v1/alerts \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "title": "Database connection pool exhausted",
    "description": "All 100 connections are in use",
    "priority": "critical",
    "source": "connection-monitor",
    "tags": ["database", "production"]
  }'

Via Webhook

For automated alerting from monitoring tools:

curl -X POST http://localhost:8081/api/v1/webhook/<token> \
  -H "Content-Type: application/json" \
  -d '{
    "title": "High memory usage on server-01",
    "description": "Memory at 95%",
    "priority": "high",
    "source": "prometheus"
  }'

Managing Alerts

Acknowledge an Alert

Acknowledging indicates someone is working on the issue:

curl -X POST http://localhost:8081/api/v1/alerts/{id}/acknowledge \
  -H "Authorization: Bearer <token>"

Snooze an Alert

Temporarily silence an alert until a specific time:

curl -X POST http://localhost:8081/api/v1/alerts/{id}/snooze \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "snooze_until": "2024-01-15T10:00:00Z"
  }'

Close an Alert

Mark an alert as resolved:

curl -X POST http://localhost:8081/api/v1/alerts/{id}/close \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "close_reason": "Issue resolved after database restart"
  }'

Assign an Alert

Assign to a specific user or team:

curl -X POST http://localhost:8081/api/v1/alerts/{id}/assign \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "assignee_id": "user-uuid",
    "assignee_type": "user"
  }'

Filtering Alerts

List alerts with various filters:

# Filter by status
GET /api/v1/alerts?status=open

# Filter by priority
GET /api/v1/alerts?priority=critical

# Filter by source
GET /api/v1/alerts?source=prometheus

# Search by title
GET /api/v1/alerts?search=database

# Combine filters
GET /api/v1/alerts?status=open&priority=critical&limit=10

Alert Fields

Field Type Description
id UUID Unique identifier
title string Alert title (required)
description string Detailed description
status string open, acknowledged, snoozed, closed
priority string critical, high, medium, low
source string Origin of the alert
tags array Labels for categorization
assigned_to UUID Assigned user or team ID
created_at timestamp When the alert was created
